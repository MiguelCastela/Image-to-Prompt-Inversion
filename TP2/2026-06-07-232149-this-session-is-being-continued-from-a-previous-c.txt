● I dug into phase4_results.json, phase4_summary.json, the per-branch winners,
  and the propose code. Here's what actually happened — and it's not really a
  "bad model" problem, it's that the loop stalled after iteration 1 and the 
  metric it optimizes can't see most of your refinements. Five concrete things
  you're missing:

  1. The loop converged at iteration 1 — iterations 2–12 did basically nothing

  Look at the CLIP best -> lines in your log: they are frozen and identical from
  iter 1 through iter 12 for every image. After iter 2, almost every branch
  prints 0 distinct proposals. So the run spent the whole night reloading Qwen +
  LCM ~24 times to produce nothing new. Your good candidates are all from
  iteration 1 (and warm = iter 0).

  Why: collect_proposals feeds the entire avoid-list back to Qwen and dedups
  both exact text and visual (dHash) duplicates. At temp 0.7 with a "improve the
  previous prompt" framing, Qwen keeps regenerating near-identical prompts →
  all filtered → 0 new → convergence guard gives up. 12 iterations bought you 
  almost nothing over 1–2. That's why everything finished at 5am for
  iteration-1-quality output.

  2. CLIP's 77-token cap is silently truncating most of your prompts

  Your log is full of truncated because CLIP can only handle sequences up to 77 
  tokens. Most warm prompts — and especially refinements that add tail detail
  like , 3d render, high-definition photograph, exact palm fronds, golden 
  hour... — overflow 77 tokens, so the CLIP scorer never sees the added words.
  You're optimizing CLIP-similarity against text CLIP can't read. The Keep each 
  prompt under 30 words rule in REFINE_SYSTEM is never enforced. This is a major
  reason scores flatline.

  3. For half the images, the clip-interrogator warm start already wins

  - 1159_29: winner is warm (0.915); refinement made it worse (0.890, 0.850).
  - 7836: winner is warm (0.805) — it's the only candidate (see #4).
  - 1159_7: best new (0.741) doesn't beat warm-level (0.761).

  The closed loop only genuinely helped the two images where the warm caption
  was wrong/weak: 1159_3 fire-girl (0.781→0.849, +0.07) and 1159_25 orange juice
  (0.901→0.937, +0.04). So refinement pays off only when the starting caption 
  is bad — otherwise you're already near the ceiling.

  4. 7836 was never refined at all — a bug

  n_candidates: 1 for 7836, and it printed 0 distinct proposals on every
  iteration including iter 1. Either Qwen's output for that image never parsed
  into a numbered list, or every proposal collapsed onto the warm prompt and got
  deduped. That image got zero benefit from the entire run.

  5. CLIP-similarity is a misleading headline and disagrees with perception

  metric_rank_correlations_spearman.clip_vs_lpips = -0.27 — anti-correlated.
  1159_29's CLIP "best" (0.915) has LPIPS 0.71 (perceptually far). So "mean CLIP
  0.85" overstates how much the renders actually look like the targets. If your
  grade/report cares about visual match, optimizing CLIP-sim is steering you
  the wrong way for some images.

  ---
  Bottom line: the run "succeeded" mechanically (crash-safe, finished) but the
  optimization did one useful step and then idled, while the objective metric
  was blind to most of the edits. The outputs are essentially iteration-1
  quality.

  If you want, I can fix the high-impact ones:
  1. Enforce a ≤~60-CLIP-token cap on prompts before render/score (and tell Qwen
  to count CLIP tokens, not words) — so refinements are actually visible to the
  scorer.
  2. Early-stop per image when an iteration adds no new distinct proposal —
  would've cut the night from ~7h to well under 1h.
  3. Make proposals actually diverge — fewer iterations (~3–4), higher/varied
  temperature, and re-grounding on the target image instead of micro-editing the
  last prompt.
  4. Fix 7836 by logging Qwen's raw output to see why it parses to nothing.
  5. Pick the real objective — if perceptual match matters, rank on

❯ i thought the cap of temperature was 1                                        

● Temperature isn't capped at 1.0 — that's a common assumption but it's not a
  hard limit. 1.0 just means "sample from the model's raw distribution
  unchanged." It's a divisor on the logits before softmax:

  - < 1.0 → sharpens the distribution (more deterministic, picks the likely
  tokens)
  - = 1.0 → the model's natural distribution
  - > 1.0 → flattens it (more spread-out, more diverse) — which is exactly what
  you want to escape a branch that keeps re-emitting the same prompt

  HuggingFace transformers.generate() accepts any positive float (the OpenAI
  API, for comparison, allows up to 2.0). So 1.2 is legal and intentional: a
  mild flatten — only ~20% past neutral. I kept it modest because much higher
  (≳1.5) is where you start getting genuinely incoherent token soup. At 1.2 you
  get more varied wordings while the prompts stay readable, and anything that
  does come out malformed scores badly and gets dropped by selection anyway.

  It only reaches 1.2 when a branch has gone stale several calls in a row (0.80
  → 0.95 → 1.10 → 1.20); a healthy branch stays at 0.80.

  If you'd rather keep it ≤ 1.0 on principle, it's one knob — P4_TEMP_MAX=1.0
  (and e.g. P4_TEMP_STEP=0.07), no code change. The tradeoff is it'll break out
  of stagnation a bit less forcefully. Want me to set the default ceiling to
  1.0, or leave it at 1.2?

❯ and the metrics are agains the target images right? but we give them the best 
  current image and the metric?                                                 

● Correct on both counts. Two separate things, and yes:

  1. Scoring — always render vs. target. Every candidate is scored by
  evaluate_candidate(target_path, render_path) — CLIP image-image, LPIPS, and
  pixel MSE, each comparing that rendered image against the target image. The
  target is never compared against anything but a render. (score_prompt →
  evaluate_candidate.)

  2. The loop feedback — target + current best render + its metrics. In the
  propose step, Qwen is shown:
  - Image 1: the target
  - Image 2: the current best render for that branch
  - Text: that render's prompt and its three scores (CLIP=… LPIPS=… MSE=…)

  Then it's asked for improved prompts. In code (collect_proposals):

  current = select_best(b, bp, 1)[0]      # branch's best candidate so far
  ...
  ut = build_user_text(..., current["prompt"], current, ...)   # current's 
  metrics go in the text
  batch = qwen_propose(model, processor, s["target_path"],
  Path(current["render"]), ut, ...)
  #                                       ^ target image      ^ current best 
  render

  So the "best current" is picked by that branch's own metric — the clip branch
  shows Qwen the render with the best CLIP, the mse branch shows the render with
  the best (lowest) MSE, etc. That's what differentiates the four branches:
  same target, but each is climbing a different metric and feeding back its own
  metric-best image.

  The new candidates Qwen proposes are then rendered and scored against the
  target (step 1), the branch picks its new best, and that becomes the "current"
  shown next iteration. Standard hill-climb: target is the fixed reference, the
  current-best render + its score is the feedback signal.
